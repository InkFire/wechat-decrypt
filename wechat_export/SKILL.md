# 微信聊天导出技能

## 执行目标

从已经解密的微信数据库中导出单聊聊天记录，产出 4 类结果：

1. 可直接阅读的纯文本聊天记录
2. 对方发送的语音样本文件
3. 说话人归属复核文件
4. 结构化元数据

默认输出目录在 `wechat_export/output/exports/contacts/{姓名}/`。

## 前置条件

1. 先确认微信数据库已经解密到 `wechat_export/output/databases/decrypted/`
2. 从 `contact/contact.db` 解析联系人真实 `username`
3. 用 `MD5(username)` 定位消息表名

```python
import hashlib
table_name = "Msg_" + hashlib.md5(username.encode()).hexdigest()
```

## 核心规则

### 1. 不要跨 DB 合并 real_sender_id

`real_sender_id` 只在当前 `message_x.db` 内有意义。

- 可以在同一个 DB 内做 sender id 归属推断
- 不要把 `message_3.db` 里的 `sender_id=4` 和 `message_9.db` 里的 `sender_id=4` 当成同一个人

### 2. 先用 status 打锚点，再处理老消息

说话人判断优先级固定为：

1. `status=2` => `我`
2. `status=4` => `她`
3. 如果当前行不是 `status=2/4`，先检查这个 `real_sender_id` 是否已经被同库内的锚点消息标记过
4. 只有没有 sender id 锚点时，才降级使用 `server_seq`

降级规则：

- `server_seq=0` => 暂时判为 `我`
- `server_seq!=0` => 暂时判为 `她`

说明：

- 老库里常见 `status=3`
- 很多 `status=3` 行不能直接靠 `server_seq` 下结论
- 更稳的做法是先看同一个 `real_sender_id` 是否在别的行出现过 `status=2` 或 `status=4`

### 3. 只按同一个 DB 判断说话人 id，命中就必须触发人工复核

如果同一个 `message_x.db` 里出现下面任一情况，必须生成人工复核材料：

1. “我”侧对应多个说话人 id，例如 `db1_13` 和 `db1_1834`
2. “她”侧对应多个说话人 id，例如 `db1_7` 和 `db1_22`
3. 某些消息只能靠 `server_seq` 推断
4. 同一个说话人 id 同时出现 `status=2` 和 `status=4`
5. 当前 DB 里只出现了一个说话人 id，需要人工确认它到底是“我”还是“她”

这里的“说话人 id”指的是 `dbx_y` 形式：

- `dbx` 表示消息来自哪个 `message_x.db`
- `y` 表示这个 DB 里的 `real_sender_id`
- 只能在同一个 DB 内比较，例如 `db1_13` 和 `db1_1834`
- 不要把 `db1_13` 和 `db9_13` 当成同一个人

复核材料必须展示样本，不能只给统计结论。

每个可疑 `sender_id` 至少展示：

- 预测说话人
- 消息条数
- `status` 分布
- `server_seq` 分布
- 3 到 5 条样本消息

目的：

- 让人工确认哪个 sender id 属于“我”
- 让人工确认哪个 sender id 属于“她”
- 防止脚本把老消息整库判反

### 4. 纯文本导出不能泄露原始 XML

纯文本聊天记录里不要直接写 `<msg>`、`<img>`、`<voicemsg>`、CDN URL、`aeskey`、`md5`、`appid` 这类原始 XML 内容。

要把非文本消息收敛成摘要：

- 文本：保留正文
- 图片：`[图片]`
- 语音：`[语音 6.9s]`
- 表情：`[表情]`
- 视频：`[视频]`
- 名片：`[名片] 昵称`
- 位置：`[位置] 地点`
- 链接/文件：`[链接] 标题 - 描述` 或 `[文件] 文件名`
- 系统消息：保留清洗后的系统文案
- 撤回消息：保留清洗后的撤回提示

不要把“纯文本导出”做成 XML 转储。

### 5. 保留对阅读有帮助的内容

清洗文本时：

- 保留中文、英文、数字、常见标点、emoji
- 保留用户输入的文本表情，如 `[旺柴]`、`[破涕为笑]`
- 把多余空白压成单空格
- 把多行内容压成可阅读的单行或少量换行

### 6. 处理内容编码

优先按下面顺序解码：

1. `WCDB_CT_message_content == 4` 且内容是字节串时，先做 zstd 解压
2. 再按 UTF-8 解码
3. 失败时允许 `errors="replace"`

不要为了“看起来没有乱码”去盲目尝试一串编码并误解正文。

## 输出要求

### 聊天记录

文件名：`{姓名}_origin.txt`

格式尽量贴近阅读体验：

```text
时间戳⟦说话人⟦说话人id⟦消息正文
2025-12-03 09:23:17⟦她⟦db0_1228⟦最近咋样呀
2025-12-03 09:24:02⟦我⟦db0_1⟦还行
2025-12-03 09:24:40⟦她⟦db0_1228⟦[图片]
2025-12-03 09:25:13⟦她⟦db0_1228⟦[语音 6.9s]
```

不要再输出：

```text
2025-12-03 09:24:40|她|db0_1228|[图片] <?xml version="1.0"?><msg><img ...>
```

### 说话人复核文件

文件名：`speaker_review.txt`

必须按 DB 分组，给出人工可判断的样本。例如：

```text
## message_1.db
触发人工复核原因:
- 同一个 DB 内，“我”侧出现多个说话人id: ['db1_13', 'db1_185']
- 同一个 DB 内存在仅靠 server_seq 推断的说话人id: ['db1_13']

说话人id db1_13 | 预测说话人 我 | 消息 97 条 | status {2: 4, 3: 93}
样本:
- 2024-10-18 10:03:11 | 我 | 文本 | 你最近怎么样
- 2024-10-18 10:03:28 | 我 | 图片 | [图片]
```

### 元数据

文件名：`metadata.json`

至少包含：

- 联系人解析结果
- 总消息数
- 对方语音数
- 各 DB 消息数
- sender id 锚点映射
- sender id 的 `status/server_seq` 统计
- `speaker_review.txt` 文件名
- 需要人工复核的 DB 数量

### 联系人配置

文件名：`contact_config.json`

这个文件是联系人自己的本地配置，不是所有联系人通用的全局配置。

适合放在这里的内容：

- 当前联系人的人工确认说话人覆盖
- 只对当前联系人成立的导出修正

当前约定：

- 路径是 `wechat_export/output/exports/contacts/{姓名}/contact_config.json`
- 人工确认覆盖写在 `manual_assignments` 里
- 说话人值允许是 `我`、`她`、`系统`
- 不要把某个联系人的人工确认结果写成全局共享规则

示例：

```json
{
  "contact_name": "英畅",
  "identifier": "yingchang001",
  "username": "yingchang001",
  "manual_assignments": {
    "db1_13": "我",
    "db1_1834": "她",
    "db10_15": "系统"
  }
}
```

### 可读版聊天记录

文件名：`{姓名}_readable.txt`

这份文件是给人直接阅读的最终文本版本，必须满足两个前提：

1. 已经重新生成过 `{姓名}_origin.txt`
2. 独立校验脚本确认当前联系人 `0` 个未确认说话人 id

生成原则：

- 来源必须是已经定稿的 `{姓名}_origin.txt`
- 不要再携带 `说话人id`
- 不要重新做数据库推断
- 不要用 LLM 改写正文
- 只用脚本做格式转换

推荐格式：

```text
[2024-11-20 10:21:08] 我: 畅畅
[2024-11-20 10:28:59] 她: 不生，我老公对孩子恐惧[捂脸]
[2015-08-19 22:47:57] 系统: 英申易开启了好友验证，你还不是他（她）好友。请先发送好友验证请求，对方验证通过后，才能聊天。发送好友验证
```

推荐脚本：

- `python wechat_export\\generate_readable_transcripts.py --contact 英畅`

### 语音样本

放在 `voice_samples/`

- 优先导出对方发送的语音
- 保留原始 `.silk` 或 `.bin`
- 文件名里带上 DB、时间、local_id、时长

## 推荐执行顺序

1. 解析联系人真实 `username`
2. 扫描所有 `message_*.db`，定位对应表
3. 逐个 DB 读取消息
4. 建立当前 DB 的 sender id 锚点
5. 生成纯文本聊天记录
6. 导出语音样本
7. 读取当前联系人的 `contact_config.json`（如果存在）
8. 写入 `speaker_review.txt` 和 `metadata.json`
9. 运行独立校验脚本 `review_speaker_assignments.py`
10. 抽查至少一个联系人的导出结果
11. 把 `review_speaker_assignments.py` 产出的样本发给人工确认
12. 人工确认完成后，重新生成 `{姓名}_origin.txt`
13. 确认当前联系人 `0` 个待确认问题
14. 运行 `generate_readable_transcripts.py` 生成 `{姓名}_readable.txt`

## 缺 key 时的时间段定位

如果提 key 或解密阶段发现某些 `message_x.db` 缺 key，不要只告诉用户“还有几个库失败了”，而要继续定位这些分库大概覆盖的时间段。

目标：

- 告诉用户缺的是哪一个 DB
- 告诉用户这个 DB 大概对应哪段历史聊天
- 让用户去微信里主动打开那段时间附近的聊天，提升把 key 刷进内存的概率

推荐流程：

1. 列出已经成功解密的 `message_*.db`
2. 逐个已解密 DB 检索消息时间跨度
3. 按 DB 编号和时间范围排序
4. 按“更老邻居 / 更新邻居”确定缺失 DB 两侧的相邻分库
5. 把推断出的时间段明确告诉用户

时间跨度的取法：

- 优先在每个 `message_x.db` 里统计消息最早时间和最晚时间
- 如果库内有多个 `Msg_*` 表，就用这个 DB 内所有消息表的全局最小值和最大值
- 时间统一转成人能直接理解的绝对时间，例如 `2016-08-13 21:36:45`

怎么推断缺失分库的时间：

补充一个非常重要的经验：

- `message_x.db` 的编号顺序通常和时间顺序相反，编号越大往往越老
- 所以不要机械地把“编号更小”当成前一个时间邻居
- 应该按时间意义区分“更老邻居”和“更新邻居”

1. 如果缺失 DB 的更老邻居和更新邻居都已解密，就用：
   更老邻居的最晚时间
   更新邻居的最早时间
   把中间这段当成“缺失 DB 大概率覆盖的时间段”
2. 如果只有更老邻居存在，就告诉用户：
   缺失 DB 大概率覆盖“晚于这个时间”的历史消息
3. 如果只有更新邻居存在，就告诉用户：
   缺失 DB 大概率覆盖“早于这个时间”的历史消息
4. 如果连续缺多个 DB，就把每个缺失 DB 单独列出来，并说明当前只能给出较粗的范围

面向用户的输出要求：

- 不要只写 `message_9.db 缺 key`
- 要写成类似：
  `message_9.db` 缺 key，结合已解密分库推断，它大概率覆盖 `2016-08-13 21:36:45` 到 `2017-08-13 11:14:30` 这段聊天
- 再补一句建议动作：
  请去微信里打开这段时间附近的老聊天，尤其是这一年内活跃的会话，再重新提 key

经验备注：

- 这类时间段提示通常比“重启微信再试一次”更有效
- 越精确到具体年月，用户越容易主动把目标聊天翻出来
- 如果已经知道某个联系人的聊天主要落在缺失时间段，也可以直接提示用户优先打开那个人的历史消息
- 可以直接运行 `python wechat_export\\estimate_missing_message_db_ranges.py`
- 如果要模拟某个缺失库，也可以运行 `python wechat_export\\estimate_missing_message_db_ranges.py --missing 9`
- 脚本会把结果写到 `wechat_export/output/reports/missing_message_db_ranges/report.txt` 和 `report.json`

## 必做校验

导出完成后，至少做这几项检查：

1. 纯文本里不应再出现大段 `<msg>`、`<img>`、`cdnurl=`、`aeskey=`、`appid=`
2. 文本消息应该以 `时间戳⟦说话人⟦说话人id⟦消息正文` 的四列形式出现
3. 对方语音样本数量要和元数据一致
4. 只要同一个 DB 内同侧出现多个说话人 id，就必须有 `speaker_review.txt`
5. 只要同一个 DB 内只出现一个说话人 id，也必须有 `speaker_review.txt`
6. 要再用独立脚本 `review_speaker_assignments.py` 生成面向人工回复的问题清单
7. 如果复核文件或独立脚本触发了复核，要把样本给人工确认，而不是自己武断定案
8. 只有当独立校验脚本显示 `0` 个待确认问题时，才允许生成 `*_readable.txt`
9. `*_readable.txt` 必须从 `*_origin.txt` 脚本化转换而来，不能重新推断说话人

## 经验备注

1. 老库常见 `status=3`，这时 sender id 锚点比 `server_seq` 更可靠
2. `message_10.db` 到 `message_13.db` 这类早期分库里，`server_seq` 可能几乎全是 0
3. 图片、表情、链接消息最容易把 XML 原文误写进纯文本，必须显式摘要化
4. 只要用户要求“纯文本聊天记录”，就优先保证可读性，而不是保留数据库原始字段
